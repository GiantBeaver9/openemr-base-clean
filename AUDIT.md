# OpenEMR Fork — Pre-Build Audit

**Purpose.** This audit is the hard-gate deliverable that precedes building the Clinical Co-Pilot AI layer. It is a grounded, code-level assessment of the OpenEMR fork the co-pilot will be embedded in — what is safe to inherit, what is a landmine, and what the AI layer must therefore own itself. Every finding cites real code (`file:line`). The findings visibly feed the design decisions recorded in [ARCHITECTURE.md](ARCHITECTURE.md), [ARCHITECTURE_COMPLETE.md](ARCHITECTURE_COMPLETE.md), and [docs/clinical-copilot-tradeoffs.md](docs/clinical-copilot-tradeoffs.md); the traceability table at the end maps each load-bearing finding to the decision it drove.

**Method.** The codebase map (`INDEX.md` / `repomap.json`, produced by a brownfield-index fan-out) was reused as the starting point. Six focused investigations then read the actual source: auth/session, ACL/authorization, output-escaping/injection, performance/schema, audit-logging/compliance, and clinical-data quality. Line citations are to this fork's tree (`sql/database.sql`, `src/`, `library/`, `interface/`, `apis/`, `ccdaservice/`).

**Severity scale.** CRITICAL (exploitable or safety-critical for an AI reading the chart), HIGH (systemic weakness, exploitable under realistic conditions), MEDIUM (real risk, bounded or config-dependent), LOW (hardening), STRENGTH (a property the co-pilot can rely on).

---

## Executive Summary

The OpenEMR fork is a mature, defensively-built monolith whose security architecture is **fail-open by design**: strong primitives exist (modern password hashing, HMAC CSRF, a real audit subsystem, envelope encryption), but their correctness depends on developers remembering to invoke them at every call site rather than on a framework enforcing them centrally. For an AI clinical co-pilot, three consequences dominate, and each one shaped the architecture.

**1. Authorization does not scope patients — so pinning must be structural.** `AclMain::aclCheckCore('patients','med')` is a *feature* gate ("may this user use Medical Records"), not a *patient* gate; it takes no patient argument ([AclMain.php:166](src/Common/Acl/AclMain.php:166)). Switching the active patient (`setpid`) is not ACL-checked ([library/pid.inc.php:16](library/pid.inc.php)), the standard REST API has no patient-compartment binding at all, and the FHIR path's `checkUserHasAccessToPatient()` is a stub returning `true` ([BearerTokenAuthorizationStrategy.php:479](src/RestControllers/Authorization/BearerTokenAuthorizationStrategy.php:479)). The co-pilot therefore cannot infer patient entitlement from the host — which is exactly why patient pinning is a structural invariant (I10), no tool accepts a patient argument, and every fact's `pid` is re-verified on output (verifier V3).

**2. The clinical data is semantically untyped — so extraction must be deterministic and verified, never delegated to the LLM.** `procedure_result.result_status` is free-text `varchar(31)` defaulting to `''` ([database.sql:10508](sql/database.sql:10508)); `result` is untyped `varchar(255)` holding numbers, `"<7.0"`, ranges, and prose ([database.sql:10503](sql/database.sql:10503)); `units` is optional free text ([database.sql:10502](sql/database.sql:10502)); corrections are UPDATE-in-place with no value history; and there are **zero enforced foreign keys anywhere** in the schema. An LLM reading these tables directly would surface wrong values, wrong units, stale/deleted data, and duplicate medications. This is the entire justification for the fact layer's lab contract (C1–C4), the "no unit, no math" rule, supersession handling, and the deterministic-capabilities-not-LLM-extraction core (T1/T14).

**3. Nothing controls PHI leaving the building — so the LLM boundary is the gate.** There is no BAA concept, no egress control, and no automatic disclosure-accounting for third-party transmission anywhere in the code. The audit log is genuinely automatic and tamper-*evident* (sha3-512 checksums) but is stored plaintext-at-rest and is not tamper-*resistant* (rows are UPDATE/DELETE-able). This validates the "synthetic patients only until a named BAA/redaction story exists" gate (OPEN-1), the module-owned trace store that keeps PHI inside the EMR's MySQL (T16), and the requirement that the module log its own patient-scoped disclosure events rather than assume inheritance.

Supporting findings reinforce the shape of the design. Twig autoescape is **globally disabled** ([TwigContainer.php:70](src/Common/Twig/TwigContainer.php:70)) with 78 unescaped sinks across 199 templates — so LLM-generated prose must be escaped explicitly at every Twig sink. The one existing side-daemon, `ccdaservice`, has a cross-patient concurrency race from mutable module-global state written after an `await` ([serveccda.js:3622](ccdaservice/serveccda.js:3622)) — the track record that argued against a sidecar (T2) and for an in-process module. Performance is favorable where it matters: per-patient reads are genuinely cheap for labs, meds, and vitals because the join columns are indexed, validating the recompute-facts-on-every-read freshness model (T5) — with two exceptions to engineer around (an unindexed calendar `pc_pid`, and an N+1 in `ProcedureService::getAll()`).

The bottom line: **the host gives the co-pilot auth, session, CSRF, audit plumbing, and cheap indexed reads for free — but it does not give patient-level authorization, typed clinical data, or any PHI-egress control. Those three gaps are precisely what the deterministic fact layer, structural patient pinning, and the LLM trust boundary in the architecture exist to close.**

## Part 1 — Security Audit

### 1A. Authentication & Session

**Strengths the co-pilot can rely on.** Password hashing is modern and self-migrating — `AuthHash` resolves `PASSWORD_DEFAULT` and supports BCRYPT/Argon2i/Argon2id/SHA512-crypt with rehash-on-login ([AuthHash.php:34](src/Common/Auth/AuthHash.php:34), [AuthUtils.php:449](src/Common/Auth/AuthUtils.php:449)). Verification and all sensitive comparisons are constant-time via `password_verify()`/`hash_equals()` ([AuthHash.php:187](src/Common/Auth/AuthHash.php:187), [AuthUtils.php:852](src/Common/Auth/AuthUtils.php:852)). Brute-force protection is two-layered (per-username and per-IP lockout with timed reset and admin notification, [AuthUtils.php:294](src/Common/Auth/AuthUtils.php:294)), timing attacks are mitigated with a persisted dummy hash ([AuthUtils.php:89](src/Common/Auth/AuthUtils.php:89)), and session fixation is handled — login calls `session->migrate(true)` ([main_screen.php:403](interface/main/main_screen.php:403)). Session cookie defaults are hardened (`use_strict_mode`, `SameSite=Strict`, 48-byte id — [SessionConfigurationBuilder.php:20](src/Common/Session/SessionConfigurationBuilder.php:20)).

Presence of `authUserID` does **not** prove MFA completed. | MEDIUM | [auth.inc.php:62](library/auth.inc.php), [main_screen.php:143](interface/main/main_screen.php:143), [AuthUtils.php:837](src/Common/Auth/AuthUtils.php:837) |

This is why §4 of ARCHITECTURE.md re-derives the principal (`authUserID`), gates every surface on ACL *and* verifies the session `user_id` matches on every turn.

### 1B. Authorization (ACL)

The ACL engine is vendored phpGACL; decisions resolve through `AclMain::aclCheckCore` with correct **deny-overrides-allow** precedence and **fail-closed** behavior on empty results ([AclMain.php:166](src/Common/Acl/AclMain.php:166), [Gacl.php:506](src/Gacl/Gacl.php:506)). FHIR patient-compartment filtering is real and well-placed ([ResourceServiceSearchTrait.php:89](src/Services/FHIR/Traits/ResourceServiceSearchTrait.php:89)). Those are genuine strengths. The gaps are about *what* ACL scopes:

| # | Finding | Severity | Location |
|---|---------|----------|----------|
| A1 | **`aclCheckCore('patients','med')` is a feature gate, not a per-patient gate** — it takes no patient argument. A `true` means "may use Medical Records," never "cleared for patient N." The active patient is an ambient `$_SESSION['pid']`. | CRITICAL (for the AI threat model) | [AclMain.php:166](src/Common/Acl/AclMain.php:166) |
| A2 | **The only record-level scoping is the opt-in "squads" mechanism** — a page must voluntarily fetch a patient's `squad` and check it; almost nothing does. No facility/care-team scoping exists. Any user with the feature permission can load any patient. | HIGH | [AclExtended.php:47](src/Common/Acl/AclExtended.php:47), [encounters.php:76](interface/patient_file/history/encounters.php:73) |
| A3 | **Page authorization is opt-in; `globals.php` authenticates but does not authorize.** No front-controller maps URL→required permission; a standalone page that omits `aclCheckCore` is reachable by any authenticated user. ~26% of interface pages call it; there is no central safety net. | HIGH | [globals.php](interface/globals.php) |
| A4 | **`setpid()` is not ACL-gated** — it just writes `$_SESSION['pid']` after an int-cast, and `globals.php` will adopt `pid` from `$_GET`/`$_POST` when the session has none. Nothing prevents the session being re-pointed at another patient. | HIGH | [library/pid.inc.php:16](library/pid.inc.php), [globals.php:770](interface/globals.php) |
| A5 | **FHIR `checkUserHasAccessToPatient()` is a stub returning `true`.** The intended access-verification backstop is a TODO; security rests entirely on the assumption that only authorized patients reach the token context. | HIGH | [BearerTokenAuthorizationStrategy.php:479](src/RestControllers/Authorization/BearerTokenAuthorizationStrategy.php:479) |
| A6 | **Standard `/api/*` REST surface has no patient-compartment binding** — `GET /api/patient/:puuid` authorizes only with feature-level `aclCheckCore`; a `user/` token can iterate any UUID. | HIGH | [_rest_routes_standard.inc.php:99](apis/routes/_rest_routes_standard.inc.php), [RestConfig.php:180](src/RestControllers/Config/RestConfig.php:180) |
| A7 | **FHIR patient-context requests bypass the ACL check entirely** (the `isPatientRequest()` branch skips it, relying on scope + the stubbed A5). ACL and scope are two parallel, non-overlapping paths. | MEDIUM | [FhirGenericRestController.php:93](src/RestControllers/FHIR/FhirGenericRestController.php:93) |
| A8 | **`admin/super` short-circuits *every* ACL check to `true`** — a single all-or-nothing key that also bypasses squads. Under an admin session, `aclCheckCore` is meaningless as a per-patient signal. | MEDIUM | [AclMain.php:173](src/Common/Acl/AclMain.php:173) |

**Co-pilot consequence.** This cluster is the direct rationale for **structural patient pinning** (I10, T14): the session's pinned `pid` is injected server-side into every tool, no tool accepts a patient argument, and the verifier re-checks every fact's `pid` (V3). The module must never infer patient-level entitlement from a feature ACL result — it binds one patient and re-verifies that binding on the way out.

## Part 2 — Performance Audit

The co-pilot's freshness model (T5) recomputes facts on **every** read and caches only the narrative, so per-patient read latency is load-bearing. The good news: it holds for four of five data sources. The findings are the two places it doesn't, plus the service-layer patterns to route around.

**Verified strength.** All eight target clinical tables are **InnoDB** (zero `ENGINE=MyISAM` in the schema) — row-level locking and transactions, no table-lock contention. Per-patient reads are genuinely index-assisted for labs, meds, vitals, and demographics, confirming the architecture's "cheap indexed per-patient read" claim: `procedure_order.patient_id` ([database.sql:10409](sql/database.sql:10408)), `prescriptions.patient_id` ([:8749](sql/database.sql:8749)), `lists.pid`/`type` ([:7709](sql/database.sql:7709)), `form_vitals.pid` ([:2449](sql/database.sql:2449)), `patient_data.pid` UNIQUE ([:8467](sql/database.sql:8467)) are all indexed. `VitalsService::search()` is the exemplar — single query, explicit columns, no N+1 ([VitalsService.php:85](src/Services/VitalsService.php:85)).

| # | Finding | Severity | Location |
|---|---------|----------|----------|
| P1 | **`openemr_postcalendar_events.pc_pid` is NOT indexed** — no index leads with `pc_pid`, so "this patient's appointments" is a full scan of the practice-wide calendar (the largest, highest-churn table), scaling with total practice size. Aggravated by `pc_pid varchar(11)` vs `patient_data.pid bigint` (implicit conversion defeats indexing). | HIGH | [database.sql:8261](sql/database.sql:8261) |
| P2 | **`ProcedureService::getAll()` is a textbook N+1** — an order loop issues per-order queries for codes and reports, then a nested per-report query for results: `1 + O·2 + O·R` queries (~100+ for 20 orders × 3 reports). Route through `search()` (single LEFT-JOIN) instead — though `search()` itself has a secondary specimen N+1. | HIGH | [ProcedureService.php:589](src/Services/ProcedureService.php:589), [:745](src/Services/ProcedureService.php:745), [:475](src/Services/ProcedureService.php:475) |
| P3 | **Per-query audit INSERT on the read path** when `audit_events_query` is enabled — every SELECT becomes read + parse + write. Combined with P2, one "get labs" turn could emit 100+ audit INSERTs. Both flags default off in standard installs, so this is a latency cliff for compliance-hardened deployments, not baseline. | MEDIUM (config) | [ADODB_mysqli_log.php:26](library/ADODB_mysqli_log.php:26), [EventAuditLogger.php:405](src/Common/Logging/EventAuditLogger.php:405) |
| P4 | **Lab labels resolve only through a 3-hop join** (`procedure_result → report → order → patient_id`; `procedure_result` has no `patient_id`). Cheap as an indexed nested-loop *if driven from `procedure_order`*; a result-code/LOINC-first query has no patient anchor and no index on `result_code` → scan. | MEDIUM | [database.sql:10493](sql/database.sql:10493) |
| P5 | **Medication UNION query applies the patient filter *above* the materialized derived table** — the `prescriptions ∪ lists` union may evaluate both branches practice-wide before filtering, and `UNION` (not `UNION ALL`) forces a dedup sort. May not benefit from the indexed `patient_id`/`pid`. Warrants `EXPLAIN` on a populated DB. | MEDIUM | [PrescriptionService.php:91](src/Services/PrescriptionService.php:91), [:205](src/Services/PrescriptionService.php:205) |
| P6 | `SELECT *` / all-column selects on wide tables — `ProcedureService` (`SELECT *` on `procedure_result`/`procedure_report`), and `BaseService::search()` effectively selects every column. `patient_data` is ~130 columns with off-page TEXT ([:8334](sql/database.sql:8334)) — costly when read directly. Select explicit narrow column lists. | MEDIUM | [ProcedureService.php:909](src/Services/ProcedureService.php:891), [BaseService.php:498](src/Services/BaseService.php:498) |
| P7 | Service constructors run `createMissingUuidsForTables` on instantiation (8 tables for `ProcedureService`) — per-instantiation work on the request path. If the module constructs services per request/patient, this repeats. | LOW–MEDIUM | [ProcedureService.php:46](src/Services/ProcedureService.php:46) |

**Co-pilot consequence.** The recompute-on-read freshness invariant (T5) is viable because P4/meds/vitals are indexed per-patient reads — but the fact layer's data access should (a) drive lab queries from `procedure_order`, (b) use `ProcedureService::search()` not `getAll()`, (c) select explicit columns, and (d) `EXPLAIN` the medication union. P1 (calendar scan) matters for the background worker's schedule sweep (`openemr_postcalendar_events` for the day's appointments, ARCHITECTURE.md worker model) — the pre-warm query needs care or an added `(pc_pid, pc_eventDate)` index at the deployment. P3 is why the module's own trace writes are append-only INSERTs to module tables, not additional load on the host `log` table.

## Part 3 — Architecture Audit

**System organization (reused from [INDEX.md](INDEX.md) / [repomap.json](repomap.json)).** OpenEMR is a PHP 8.2+ monolith mid-modernization: a modern PSR-4 layer (`src/`, `OpenEMR\`) of ~100 `BaseService` domain services, REST/FHIR controllers, and a Symfony kernel/event system, sitting on a legacy procedural layer (`library/`) and ~590 server-rendered pages (`interface/`) that each bootstrap through `interface/globals.php`. Around it: a patient portal (`portal/`) with its own session, two plugin systems (`interface/modules/`), a Node C-CDA side-service (`ccdaservice/`), Twig/Smarty templates, a dual schema mechanism (`sql/` live, `db/` Doctrine not yet wired), and a webpack asset pipeline.

**Where clinical data lives** (the co-pilot's read surface, all read-only to the module): labs in the `procedure_order → procedure_report → procedure_result` triple; medications split across `prescriptions` and `lists (type='medication')`; vitals in `form_vitals`; demographics in `patient_data`; appointments in `openemr_postcalendar_events`. Full schema landmines in Part 4.

**Layer interactions & blast radius.** The core coupling is `src ⇄ library` (library increasingly delegates down to `src`; `src` still `require`s legacy `.inc.php` files), and **everything bootstraps through `interface/globals.php`** — the single point that establishes session, site, DB, auth, the globals bag, and module boot. Per repomap's blast-radius graph, a change in `globals.php` reaches interface-modules, portal, tests, the ccdaservice gateway, and root entry points. `portal/` is a leaf (nothing depends on it); `interface-modules` is loaded at bootstrap by interface-core and is otherwise leaf-ish — which is exactly the low-collision surface an additive custom module wants.

**Integration points for the AI layer** (all sanctioned host extension mechanisms the co-pilot reuses):
- **Custom module system** — `interface/modules/custom_modules/oe-module-*/` with `openemr.bootstrap.php` registering a PSR-4 namespace and subscribing `src/Bootstrap.php` to core events ([ModulesApplication.php](src/Core/ModulesApplication.php)). This is the co-pilot's placement (T2), and additivity is enforceable (I9): empty diff outside the module dir, byte-identical host when disabled, clean uninstall.
- **Service layer** — `BaseService` subclasses (notably `PrescriptionService` for the meds union, T4) give the fact layer typed, tested reads without re-implementing joins.
- **`background_services` framework** — DB-scheduled worker with `next_run`/`execute_interval`/lease-lock, reused as the pre-warm worker (T11) — no new daemon (consistent with T2's no-sidecar stance).
- **`EventAuditLogger`** — inherited PHI audit trail for chart reads (Part 5, and ARCHITECTURE.md §1.3).
- **Symfony EventDispatcher** — modules extend core by subscribing, never by patching; the basis for optional warm-hints (T5 extension point).
- **`QueryUtils`, `OEGlobalsBag`, `CsrfUtils`, Twig, `HealthCheckInterface`** — host-provided infrastructure the module composes rather than reinvents.

**Architectural risk the audit surfaces.** The monolith's extension points are strong, but its cross-cutting enforcement is decentralized (auth per-page A3, CSRF per-handler X5, escaping per-sink X1, audit per-flow Part 5). A new capability inherits *plumbing* for free but inherits *no enforcement guarantees* — the module must re-assert each control at its own boundaries. This is why the architecture makes verification a deterministic gate (§2) and additivity an enforced invariant (I9/T12) rather than trusting the host's ambient behavior.

## Part 4 — Data Quality Audit

Every clinical table examined is InnoDB but **not one declares a `FOREIGN KEY`/`CONSTRAINT`/`REFERENCES`** (0 across `sql/database.sql`); "references" appears only in advisory column comments. Controlled vocabularies, referential links, and row liveness are all enforced in application code, never in the schema — so an AI reading the tables directly inherits all of it as untrusted input. These findings are the empirical basis for the lab contract (C1–C4), the digest/supersession rules, and "no unit, no math" (T8/T9).

| # | Finding | Severity | Location & agent-failure mode |
|---|---------|----------|-------------------------------|
| D1 | **`result_status` is free-text `varchar(31)` default `''`** — comment's own "…etc." advertises an open vocabulary (`final`, `preliminary`, `corrected`, `cannot be done`, `''`, plus whatever HL7/manual entry writes). No enum, no FK. `procedure_report.report_status` is a second free-text status layer. **Failure:** can't distinguish final from preliminary/superseded; `''` default is indistinguishable from "unknown." | CRITICAL | [database.sql:10508](sql/database.sql:10508) |
| D2 | **`result` is untyped `varchar(255)`** — holds numerics, censored values (`"<7.0"`), ranges, qualitative text, `''` — all in one column. **Failure:** naive parse coerces `"<7.0"` → `7.0` (erasing the censoring), blank → `0`. Wrong-value on every quantitative trend/threshold. | CRITICAL | [database.sql:10503](sql/database.sql:10503) |
| D3 | **`units` is free-text `varchar(31)` default `''`**; the `procedure_type.units` fallback is equally optional. **Failure:** A1c `"6.5"` with no unit is ambiguous `%` (NGSP) vs `mmol/mol` (IFCC) — an order of magnitude apart; `"mg/dl"`/`"mg/dL"`/`"MG/DL"` don't compare equal. | HIGH | [database.sql:10502](sql/database.sql:10502) |
| D4 | **In-place corrections with no value history.** `procedure_result` has no `date_modified`/row-version/`activity` at all; `form_vitals.last_updated`, `prescriptions.date_modified`, `lists.modifydate` record *when* but overwrite the prior *value*. Corrections are UPDATE-in-place. **Failure:** can't detect a value it reasoned over was corrected, nor reconstruct the value as of a past decision (a critical→normal correction shows only the new value). | CRITICAL | [database.sql:10493](sql/database.sql:10493), [:2447](sql/database.sql:2447), [:8705](sql/database.sql:8704), [:7704](sql/database.sql:7704) |
| D5 | **`result_data_type char(1)` defaults to `'S'` (String)** — values `N/S/F/E/L`, not FK-constrained, safe default is "text." Rows inserted without setting it claim to be text even when `result` is numeric. **Failure:** an agent gating numeric parse on `='N'` under-reads real numbers; `F/E/L` give no parse contract. Not a reliable numeric-vs-text discriminator. | HIGH | [database.sql:10497](sql/database.sql:10497) |
| D6 | **Medications duplicated across `prescriptions` and `lists (type='medication')`.** `lists_medication.prescription_id` is a *nullable, app-maintained, un-FK'd* link between the two; `is_primary_record` (nullable) is the only authority hint. **Failure:** reading both without reconciling on the (unenforced) link double-counts a drug; reading one misses meds present only in the other. Directly in the med-reconciliation path. | HIGH | [database.sql:7671](sql/database.sql:7671), [:7728](sql/database.sql:7716), [:8698](sql/database.sql:8698) |
| D7 | **Soft-delete via `activity`/`active` flags.** `procedure_order.activity` (`0 if deleted`), `prescriptions.active`, and `lists.activity` is **nullable with no default** (a third "unknown" state). `procedure_result` has no flag — its liveness is inherited transitively through the unenforced join. **Failure:** any query not filtering `activity=1` surfaces deleted orders/discontinued meds as current; a naive `WHERE activity=1` on `lists` silently drops NULL rows. | CRITICAL | [database.sql:10380](sql/database.sql:10380), [:8725](sql/database.sql:8725), [:7688](sql/database.sql:7688) |
| D8 | **Zero enforced referential integrity** — all inter-table links are `KEY` indexes + prose comments (`procedure_result.procedure_report_id NOT NULL` guarantees a value is present, not that it resolves). All InnoDB, so this is a choice, not an engine limit. **Failure:** joins can return orphaned/dropped rows; without the enforced report→order→patient chain, the agent cannot trust a `procedure_result` even belongs to the patient it queried. Underlies every other finding. | CRITICAL | file-wide (0 FK) |

**Co-pilot consequence.** This part *is* the fact layer's reason for existing. D1 → C2 status semantics + supersession; D2 → C3 value parsing with comparator censoring; D3 → C4 "no unit, no math"; D4 → content-addressed freshness by digest (T5) since no mutation-tracking column can be trusted; D6 → the meds UNION reconciliation (T4); D7 → `activity=1`-only slices with visible exclusion accounting (I5); D8 → the deterministic 3-table join with explicit pid re-verification (V3). The meta-invariant "no silent exclusion" (I5) is the direct answer to D5/D7: every filtered row surfaces as a cited "N excluded (reason)" fact rather than a disappearance.

## Part 5 — Compliance & Regulatory Audit

**Strengths — a genuinely mature audit subsystem.** `EventAuditLogger` captures PHI access automatically at the DB layer: the ADODB wrapper calls `auditSQLEvent()` on *every* statement, classified via the `LOG_TABLES` map ([ADODB_mysqli_log.php:26](library/ADODB_mysqli_log.php:26), [EventAuditLogger.php:107](src/Common/Logging/EventAuditLogger.php:107)) — the correct architecture for §164.312(b) audit controls. Chart VIEW is logged automatically on patient-context switch ([PatientSessionUtil.php:80](src/Common/Session/PatientSessionUtil.php:80)), so **ARCHITECTURE.md's claim that chart reads are audit-logged holds for session-based UI flows.** Entries are tamper-*evident* via sha3-512 checksums with a dedicated tamper report that even detects deleted rows ([LogTablesSink.php:63](src/Common/Logging/Audit/LogTablesSink.php:63), [audit_log_tamper_report.php:231](interface/reports/audit_log_tamper_report.php)). A pluggable `AtnaSink` emits RFC-3881/ATNA over TLS; break-glass users are logged even when auditing is disabled; the audit DB connection is isolated; and `CryptoGen` is real envelope encryption (AES-256-CBC + HMAC-SHA384, versioned keys — [CryptoGen.php:193](src/Common/Crypto/CryptoGen.php:193)). A manual accounting-of-disclosures feature (§164.528) exists ([disclosure_full.php](interface/patient_file/summary/disclosure_full.php)).

| # | Finding | HIPAA concept | Severity | Location |
|---|---------|---------------|----------|----------|
| R1 | **Audit log is NOT encrypted at rest** — the encryption path was removed (#12118/#12120); rows write `'encrypt'=>'No'` and comments are only `base64_encode`d. `log.comments`/`user_notes` and `api_log.request_body`/`response` (longtext) store PHI-bearing SQL, bound params, and full API bodies in plaintext. | §164.312(a)(2)(iv) | MEDIUM | [LogTablesSink.php:87](src/Common/Logging/Audit/LogTablesSink.php:87), [EventAuditLogger.php:660](src/Common/Logging/EventAuditLogger.php:660), [database.sql:12560](sql/database.sql:12560) |
| R2 | **Audit rows are UPDATE/DELETE-able — tamper-evident, not tamper-resistant.** No DB trigger or app control prevents alteration; the tamper report is detective, run on demand. The disclosure API itself exposes `updateRecordedDisclosure()`/`deleteDisclosure()`, and `extended_log` carries **no** checksum. | §164.312(b),(c)(1) | MEDIUM–HIGH | [EventAuditLogger.php:596](src/Common/Logging/EventAuditLogger.php:596), [database.sql:7758](sql/database.sql:7758) |
| R3 | **No retention/rotation/purge or 6-year enforcement in code** — audit tables grow unbounded (retention left to DBA backups), and `getEvents()` hard-caps at `LIMIT 5000` so older records fall out of UI review. Not violated (nothing auto-deletes), but not codified either. | §164.316(b)(2) | MEDIUM | [EventAuditLogger.php:353](src/Common/Logging/EventAuditLogger.php:353) |
| R4 | **Accounting of disclosures is manual and separate from the audit trail** — `recordDisclosure()` exists but nothing auto-captures a third-party transmission as a disclosure. `extended_log` is freely mutable and unchecksummed. | §164.528 | LOW–MEDIUM | [EventAuditLogger.php:567](src/Common/Logging/EventAuditLogger.php:567) |
| R5 | **Automatic chart-access logging has a caveat: `pid` attribution is session-inferred.** `auditSQLEvent` attaches `pid` from `session->get('pid')`, often unset in stateless API/service contexts → the row logs with `patient_id=0`, breaking "who accessed this patient." Logging is also gated by operator config (`enable_auditlog`, `audit_events_*`). | §164.312(b) | Strength w/ caveat | [EventAuditLogger.php:509](src/Common/Logging/EventAuditLogger.php:508) |
| R6 | **Key management is solid but keys are co-located with data** — drive keys on the same filesystem, DB keys in the `keys` table (envelope-encrypted, so neither alone yields plaintext), but no HSM/KMS and **no key-rotation primitive** (versioning supports decryption, nothing re-encrypts). | §164.312(a)(2)(iv) | LOW–MEDIUM | [CryptoGen.php:373](src/Common/Crypto/CryptoGen.php:373) |
| R7 | **No BAA / third-party transmission controls exist anywhere** — no egress allow-list, no outbound-PHI gate, no BAA/consent semantics. Host controls stop at the system boundary; everything is internal-access logging. Sending PHI to an external LLM is a disclosure to a business associate the host does nothing to record, gate, or govern. | §164.502(e), §164.528 | HIGH (for co-pilot) | absence |

**Co-pilot consequence.** R7 is the decisive compliance finding: **the module cannot inherit BAA/egress control because it does not exist.** This is why the architecture makes the LLM provider the single named trust boundary (§4), gates real-PHI deployment behind a named redaction/BAA story and uses **synthetic patients only** in this phase (OPEN-1), and keeps PHI-laden trace payloads inside the EMR's own MySQL rather than a third-party observability SaaS (T16). R1/R2 (plaintext, tamper-evident-only audit) reinforce keeping the module's provenance ledger and trace store in the same protection domain and treating `api_log` PHI as a live exposure. R4/R5 mean the module must **log its own patient-scoped access/transmission events with an explicit `patient_id`** (never rely on session `pid` inference) and record each external-LLM send as a disclosure — exactly the EventAuditLogger integration ARCHITECTURE.md §1.3 specifies, made concrete by these gaps. R3/R6 (no retention enforcement, no key rotation) map to OPEN-2's export-then-purge tooling as an unbuilt gate before any real-PHI deployment.

**Overall verdict.** The fork is a sound host to build on: strong crypto and hashing, a real audit subsystem, cheap indexed per-patient reads, and sanctioned in-process extension points. Its weaknesses are consistent and predictable — a fail-open posture that provides plumbing but not enforcement, clinical data that is semantically untyped, and zero controls on PHI leaving the system. The Clinical Co-Pilot's architecture is a direct response to that profile: deterministic extraction over untyped data, a verification gate the host doesn't provide, structural patient pinning the ACL layer can't guarantee, and a hard BAA/synthetic-data boundary at the one point PHI would leave the building.
