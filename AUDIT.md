# OpenEMR Fork ‚Äî Pre-Build Audit

**Purpose.** This audit is the hard-gate deliverable that precedes building the Clinical Co-Pilot AI layer. It is a grounded, code-level assessment of the OpenEMR fork the co-pilot will be embedded in ‚Äî what is safe to inherit, what is a landmine, and what the AI layer must therefore own itself. Every finding cites real code (`file:line`). The findings visibly feed the design decisions recorded in [ARCHITECTURE.md](ARCHITECTURE.md), [ARCHITECTURE_COMPLETE.md](ARCHITECTURE_COMPLETE.md), and [docs/clinical-copilot-tradeoffs.md](docs/clinical-copilot-tradeoffs.md); the traceability table at the end maps each load-bearing finding to the decision it drove.

**Method.** The codebase map (`INDEX.md` / `repomap.json`, produced by a brownfield-index fan-out) was reused as the starting point. Six focused investigations then read the actual source: auth/session, ACL/authorization, output-escaping/injection, performance/schema, audit-logging/compliance, and clinical-data quality. Line citations are to this fork's tree (`sql/database.sql`, `src/`, `library/`, `interface/`, `apis/`, `ccdaservice/`).

**Severity scale.** CRITICAL (exploitable or safety-critical for an AI reading the chart), HIGH (systemic weakness, exploitable under realistic conditions), MEDIUM (real risk, bounded or config-dependent), LOW (hardening), STRENGTH (a property the co-pilot can rely on).

---

## Executive Summary

The OpenEMR fork is a mature, defensively-built monolith whose security architecture is **fail-open by design**: strong primitives exist (modern password hashing, HMAC CSRF, a real audit subsystem, envelope encryption), but their correctness depends on developers remembering to invoke them at every call site rather than on a framework enforcing them centrally. For an AI clinical co-pilot, three consequences dominate, and each one shaped the architecture.

**1. Authorization does not scope patients ‚Äî so pinning must be structural.** `AclMain::aclCheckCore('patients','med')` is a *feature* gate ("may this user use Medical Records"), not a *patient* gate; it takes no patient argument ([AclMain.php:166](src/Common/Acl/AclMain.php:166)). Switching the active patient (`setpid`) is not ACL-checked ([library/pid.inc.php:16](library/pid.inc.php)), the standard REST API has no patient-compartment binding at all, and the FHIR path's `checkUserHasAccessToPatient()` is a stub returning `true` ([BearerTokenAuthorizationStrategy.php:479](src/RestControllers/Authorization/BearerTokenAuthorizationStrategy.php:479)). The co-pilot therefore cannot infer patient entitlement from the host ‚Äî which is exactly why patient pinning is a structural invariant (I10), no tool accepts a patient argument, and every fact's `pid` is re-verified on output (verifier V3).

**2. The clinical data is semantically untyped ‚Äî so extraction must be deterministic and verified, never delegated to the LLM.** `procedure_result.result_status` is free-text `varchar(31)` defaulting to `''` ([database.sql:10508](sql/database.sql:10508)); `result` is untyped `varchar(255)` holding numbers, `"<7.0"`, ranges, and prose ([database.sql:10503](sql/database.sql:10503)); `units` is optional free text ([database.sql:10502](sql/database.sql:10502)); corrections are UPDATE-in-place with no value history; and there are **zero enforced foreign keys anywhere** in the schema. An LLM reading these tables directly would surface wrong values, wrong units, stale/deleted data, and duplicate medications. This is the entire justification for the fact layer's lab contract (C1‚ÄìC4), the "no unit, no math" rule, supersession handling, and the deterministic-capabilities-not-LLM-extraction core (T1/T14).

**3. Nothing controls PHI leaving the building ‚Äî so the LLM boundary is the gate.** There is no BAA concept, no egress control, and no automatic disclosure-accounting for third-party transmission anywhere in the code. The audit log is genuinely automatic and tamper-*evident* (sha3-512 checksums) but is stored plaintext-at-rest and is not tamper-*resistant* (rows are UPDATE/DELETE-able). This validates the "synthetic patients only until a named BAA/redaction story exists" gate (OPEN-1) ‚Äî and shapes the *first shipped piece* of that story, **egress redaction** (identifiers tokenized before the LLM call, re-hydrated post-verification, ¬ß4/T18) ‚Äî plus the module-owned trace store that keeps PHI inside the EMR's MySQL (T16), and the requirement that the module log its own patient-scoped disclosure events rather than assume inheritance.

Supporting findings reinforce the shape of the design. Twig autoescape is **globally disabled** ([TwigContainer.php:70](src/Common/Twig/TwigContainer.php:70)) with 78 unescaped sinks across 199 templates ‚Äî so LLM-generated prose must be escaped explicitly at every Twig sink. The one existing side-daemon, `ccdaservice`, has a cross-patient concurrency race from mutable module-global state written after an `await` ([serveccda.js:3622](ccdaservice/serveccda.js:3622)) ‚Äî the track record that argued against a sidecar (T2) and for an in-process module. Performance is favorable where it matters: per-patient reads are genuinely cheap for labs, meds, and vitals because the join columns are indexed, validating the recompute-facts-on-every-read freshness model (T5) ‚Äî with two exceptions to engineer around (an unindexed calendar `pc_pid`, and an N+1 in `ProcedureService::getAll()`).

The bottom line: **the host gives the co-pilot auth, session, CSRF, audit plumbing, and cheap indexed reads for free ‚Äî but it does not give patient-level authorization, typed clinical data, or any PHI-egress control. Those three gaps are precisely what the deterministic fact layer, structural patient pinning, and the LLM trust boundary in the architecture exist to close.**

## Part 1 ‚Äî Security Audit

### 1A. Authentication & Session

**Strengths the co-pilot can rely on.** Password hashing is modern and self-migrating ‚Äî `AuthHash` resolves `PASSWORD_DEFAULT` and supports BCRYPT/Argon2i/Argon2id/SHA512-crypt with rehash-on-login ([AuthHash.php:34](src/Common/Auth/AuthHash.php:34), [AuthUtils.php:449](src/Common/Auth/AuthUtils.php:449)). Verification and all sensitive comparisons are constant-time via `password_verify()`/`hash_equals()` ([AuthHash.php:187](src/Common/Auth/AuthHash.php:187), [AuthUtils.php:852](src/Common/Auth/AuthUtils.php:852)). Brute-force protection is two-layered (per-username and per-IP lockout with timed reset and admin notification, [AuthUtils.php:294](src/Common/Auth/AuthUtils.php:294)), timing attacks are mitigated with a persisted dummy hash ([AuthUtils.php:89](src/Common/Auth/AuthUtils.php:89)), and session fixation is handled ‚Äî login calls `session->migrate(true)` ([main_screen.php:403](interface/main/main_screen.php:403)). Session cookie defaults are hardened (`use_strict_mode`, `SameSite=Strict`, 48-byte id ‚Äî [SessionConfigurationBuilder.php:20](src/Common/Session/SessionConfigurationBuilder.php:20)).

### 1B. Authorization (ACL)

The ACL engine is vendored phpGACL; decisions resolve through `AclMain::aclCheckCore` with correct **deny-overrides-allow** precedence and **fail-closed** behavior on empty results ([AclMain.php:166](src/Common/Acl/AclMain.php:166), [Gacl.php:506](src/Gacl/Gacl.php:506)). FHIR patient-compartment filtering is real and well-placed ([ResourceServiceSearchTrait.php:89](src/Services/FHIR/Traits/ResourceServiceSearchTrait.php:89)). Those are genuine strengths. The gaps are about *what* ACL scopes:

| # | Finding | Severity | Location |
|---|---------|----------|----------|
| A1 | **`aclCheckCore('patients','med')` is a feature gate, not a per-patient gate** ‚Äî it takes no patient argument. A `true` means "may use Medical Records," never "cleared for patient N." The active patient is an ambient `$_SESSION['pid']`. | CRITICAL (for the AI threat model) | [AclMain.php:166](src/Common/Acl/AclMain.php:166) |
| A2 | **The only record-level scoping is the opt-in "squads" mechanism** ‚Äî a page must voluntarily fetch a patient's `squad` and check it; almost nothing does. | A5 | **FHIR `checkUserHasAccessToPatient()` is a stub returning `true`.** The intended access-verification backstop is a TODO; security rests entirely on the assumption that only authorized patients reach the token context. | HIGH | [BearerTokenAuthorizationStrategy.php:479](src/RestControllers/Authorization/BearerTokenAuthorizationStrategy.php:479) |

## Part 2 ‚Äî Performance Audit

The co-pilot's freshness model (T5) recomputes facts on **every** read and caches only the narrative, so per-patient read latency is load-bearing. The good news: it holds for four of five data sources. The findings are the two places it doesn't, plus the service-layer patterns to route around.

**Verified strength.** All eight target clinical tables are **InnoDB** (zero `ENGINE=MyISAM` in the schema) ‚Äî row-level locking and transactions, no table-lock contention. Per-patient reads are genuinely index-assisted for labs, meds, vitals, and demographics, confirming the architecture's "cheap indexed per-patient read" claim: `procedure_order.patient_id` ([database.sql:10409](sql/database.sql:10408)), `prescriptions.patient_id` ([:8749](sql/database.sql:8749)), `lists.pid`/`type` ([:7709](sql/database.sql:7709)), `form_vitals.pid` ([:2449](sql/database.sql:2449)), `patient_data.pid` UNIQUE ([:8467](sql/database.sql:8467)) are all indexed. `VitalsService::search()` is the exemplar ‚Äî single query, explicit columns, no N+1 ([VitalsService.php:85](src/Services/VitalsService.php:85)).

| # | Finding | Severity | Location |
|---|---------|----------|----------|

Aggravated by `pc_pid varchar(11)` vs `patient_data.pid bigint` (implicit conversion defeats indexing). | HIGH | [database.sql:8261](sql/database.sql:8261) |

P1 (calendar scan) matters for the background worker's schedule sweep (`openemr_postcalendar_events` for the day's appointments, ARCHITECTURE.md worker model) ‚Äî the pre-warm query needs care or an added `(pc_pid, pc_eventDate)` index at the deployment. P3 is why the module's own trace writes are append-only INSERTs to module tables, not additional load on the host `log` table.

## Part 3 ‚Äî Architecture Audit

**System organization (reused from [INDEX.md](INDEX.md) / [repomap.json](repomap.json)).** OpenEMR is a PHP 8.2+ monolith mid-modernization: a modern PSR-4 layer (`src/`, `OpenEMR\`) of ~100 `BaseService` domain services, REST/FHIR controllers, and a Symfony kernel/event system, sitting on a legacy procedural layer (`library/`) and ~590 server-rendered pages (`interface/`) that each bootstrap through `interface/globals.php`. Around it: a patient portal (`portal/`) with its own session, two plugin systems (`interface/modules/`), a Node C-CDA side-service (`ccdaservice/`), Twig/Smarty templates, a dual schema mechanism (`sql/` live, `db/` Doctrine not yet wired), and a webpack asset pipeline.

**Where clinical data lives** (the co-pilot's read surface, all read-only to the module): labs in the `procedure_order ‚Üí procedure_report ‚Üí procedure_result` triple; medications split across `prescriptions` and `lists (type='medication')`; vitals in `form_vitals`; demographics in `patient_data`; appointments in `openemr_postcalendar_events`. Full schema landmines in Part 4.

**Layer interactions & blast radius.** The core coupling is `src ‚áÑ library` (library increasingly delegates down to `src`; `src` still `require`s legacy `.inc.php` files), and **everything bootstraps through `interface/globals.php`** ‚Äî the single point that establishes session, site, DB, auth, the globals bag, and module boot. Per repomap's blast-radius graph, a change in `globals.php` reaches interface-modules, portal, tests, the ccdaservice gateway, and root entry points. `portal/` is a leaf (nothing depends on it); `interface-modules` is loaded at bootstrap by interface-core and is otherwise leaf-ish ‚Äî which is exactly the low-collision surface an additive custom module wants.

**Integration points for the AI layer** (all sanctioned host extension mechanisms the co-pilot reuses):
- **Custom module system** ‚Äî `interface/modules/custom_modules/oe-module-*/` with `openemr.bootstrap.php` registering a PSR-4 namespace and subscribing `src/Bootstrap.php` to core events ([ModulesApplication.php](src/Core/ModulesApplication.php)). This is the co-pilot's placement (T2), and additivity is enforceable (I9): empty diff outside the module dir, byte-identical host when disabled, clean uninstall.
- **Service layer** ‚Äî `BaseService` subclasses (notably `PrescriptionService` for the meds union, T4) give the fact layer typed, tested reads without re-implementing joins.
- **`background_services` framework** ‚Äî DB-scheduled worker with `next_run`/`execute_interval`/lease-lock, reused as the pre-warm worker (T11) ‚Äî no new daemon (consistent with T2's no-sidecar stance).
- **`EventAuditLogger`** ‚Äî inherited PHI audit trail for chart reads (Part 5, and ARCHITECTURE.md ¬ß1.3).
- **Symfony EventDispatcher** ‚Äî modules extend core by subscribing, never by patching; the basis for optional warm-hints (T5 extension point).
- **`QueryUtils`, `OEGlobalsBag`, `CsrfUtils`, Twig** ‚Äî host-provided infrastructure the module composes rather than reinvents. (The host's `HealthCheckInterface`/`meta/health` prober is *not* extensible without a core diff, so the module's `/copilot/health` and `/copilot/ready` stand alone rather than registering with it ‚Äî ARCHITECTURE.md ¬ß3.4.)

**Architectural risk the audit surfaces.** The monolith's extension points are strong, but its cross-cutting enforcement is decentralized (auth per-page A3, CSRF per-handler X5, escaping per-sink X1, audit per-flow Part 5). A new capability inherits *plumbing* for free but inherits *no enforcement guarantees* ‚Äî the module must re-assert each control at its own boundaries. This is why the architecture makes verification a deterministic gate (¬ß2) and additivity an enforced invariant (I9/T12) rather than trusting the host's ambient behavior.

## Part 4 ‚Äî Data Quality Audit

Every clinical table examined is InnoDB but **not one declares a `FOREIGN KEY`/`CONSTRAINT`/`REFERENCES`** (0 across `sql/database.sql`); "references" appears only in advisory column comments. Controlled vocabularies, referential links, and row liveness are all enforced in application code, never in the schema ‚Äî so an AI reading the tables directly inherits all of it as untrusted input. These findings are the empirical basis for the lab contract (C1‚ÄìC4), the digest/supersession rules, and "no unit, no math" (T8/T9).

| # | Finding | Severity | Location & agent-failure mode |
|---|---------|----------|-------------------------------|

## Part 5 ‚Äî Compliance & Regulatory Audit

**Strengths ‚Äî a genuinely mature audit subsystem.** `EventAuditLogger` captures PHI access automatically at the DB layer: the ADODB wrapper calls `auditSQLEvent()` on *every* statement, classified via the `LOG_TABLES` map ([ADODB_mysqli_log.php:26](library/ADODB_mysqli_log.php:26), [EventAuditLogger.php:107](src/Common/Logging/EventAuditLogger.php:107)) ‚Äî the correct architecture for ¬ß164.312(b) audit controls. Chart VIEW is logged automatically on patient-context switch ([PatientSessionUtil.php:80](src/Common/Session/PatientSessionUtil.php:80)), so **ARCHITECTURE.md's claim that chart reads are audit-logged holds for session-based UI flows.** Entries are tamper-*evident* via sha3-512 checksums with a dedicated tamper report that even detects deleted rows ([LogTablesSink.php:63](src/Common/Logging/Audit/LogTablesSink.php:63), [audit_log_tamper_report.php:231](interface/reports/audit_log_tamper_report.php)). A pluggable `AtnaSink` emits RFC-3881/ATNA over TLS; break-glass users are logged even when auditing is disabled; the audit DB connection is isolated; and `CryptoGen` is real envelope encryption (AES-256-CBC + HMAC-SHA384, versioned keys ‚Äî [CryptoGen.php:193](src/Common/Crypto/CryptoGen.php:193)). A manual accounting-of-disclosures feature (¬ß164.528) exists ([disclosure_full.php](interface/patient_file/summary/disclosure_full.php)).

**Co-pilot consequence.** R7 is the decisive compliance finding: **the module cannot inherit BAA/egress control because it does not exist.** This is why the architecture makes the LLM provider the single named trust boundary (¬ß4), gates real-PHI deployment behind a named redaction/BAA story and uses **synthetic patients only** in this phase (OPEN-1), and keeps PHI-laden trace payloads inside the EMR's own MySQL rather than a third-party observability SaaS (T16). R1/R2 (plaintext, tamper-evident-only audit) reinforce keeping the module's provenance ledger and trace store in the same protection domain and treating `api_log` PHI as a live exposure. R4/R5 mean the module must **log its own patient-scoped access/transmission events with an explicit `patient_id`** (never rely on session `pid` inference) and record each external-LLM send as a disclosure ‚Äî exactly the EventAuditLogger integration ARCHITECTURE.md ¬ß1.3 specifies, made concrete by these gaps. R3/R6 (no retention enforcement, no key rotation) map to OPEN-2's export-then-purge tooling as an unbuilt gate before any real-PHI deployment.

**Overall verdict.** The fork is a sound host to build on: strong crypto and hashing, a real audit subsystem, cheap indexed per-patient reads, and sanctioned in-process extension points. Its weaknesses are consistent and predictable ‚Äî a fail-open posture that provides plumbing but not enforcement, clinical data that is semantically untyped, and zero controls on PHI leaving the system. The Clinical Co-Pilot's architecture is a direct response to that profile: deterministic extraction over untyped data, a verification gate the host doesn't provide, structural patient pinning the ACL layer can't guarantee, and a hard BAA/synthetic-data boundary at the one point PHI would leave the building.
